# Filing a Query answer is now an automatic y/n prompt, not an LLM-decided tool

Amends [ADR 0006](0006-human-gated-ingest.md): filing the last `query_wiki` answer into the wiki is still human-gated and still a deterministic wrap-with-frontmatter write, but the trigger changed. Previously the human had to know to say something like "file that answer" in natural language, and the Router LLM decided whether that counted and picked a path/title/description by calling a `file_answer` tool. Now the REPL asks `Save this answer to the wiki? [y/n]:` right after every `query_wiki` answer prints; `y`/`yes` files it, anything else (including a blank line) moves on. The `file_answer` Router tool is removed — the y/n prompt is the only way to file an answer.

The write itself is unchanged: `okf.wrap_frontmatter` on the answer text verbatim, no second LLM pass. What changed is only *who decides to ask* — a fixed REPL step instead of the LLM's tool-choice judgment — and *how the page is named* — deterministically slugified from the question (`wiki/query-answers/<slug>.md`, title = the question) instead of an LLM-picked path/title/description. Scope stays `query_wiki` only: `ingest_source` and `lint_wiki` already write to the wiki as part of doing their job, so there's no separate "answer" to offer filing.

Rejected keeping `file_answer` alongside the new prompt (e.g. for "actually, file that answer from three turns ago"): two overlapping paths to the same action is exactly the kind of dual-path drift this project's ADRs avoid elsewhere, and `last_answer`/`last_question` only ever track the most recent query anyway, so the tool couldn't reach further back than the y/n prompt already does. Rejected letting the LLM pick the path/title as before: it required a second tool round-trip for a decision (naming) that doesn't need judgment — a deterministic slug of the question is sufficient at demo scale and keeps the y/n prompt to one keystroke.
