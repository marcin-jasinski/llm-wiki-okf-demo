# Query answer links are rewritten per-backend via `Store.page_url()`

A Query answer is rendered to a one-off temp HTML file and opened via `file://` (`agent.render_answer_html`). The system prompt tells the LLM to cross-link wiki pages using the OKF bundle-relative convention, `[title](/path/page.md)` — the same syntax used *inside* wiki source files. Left as-is in a standalone rendered answer, that link is dead: `/path/page.md` resolves against the filesystem root, not `WIKI_DIR`, and even a correctly-rooted link would point at raw markdown, which browsers don't render. On the xWiki backend it's worse — there is no local file behind the link at all, so no `file://` href could ever be correct.

The fix adds `page_url(rel) -> str | None` to the `Store` interface (the same seam [ADR 0011](0011-pluggable-wiki-store.md) already uses for local/remote divergence): `LocalStore.page_url` always returns `None` ("nothing live — render me instead"); `XWikiStore.page_url` returns the page's real view URL (`XWIKI_BASE_URL/bin/view/<nested spaces>/<page>`, following [ADR 0012](0012-okf-on-xwiki-mapping.md)'s path↔space mapping). `render_answer_html` walks the whole store on every query: pages with a `page_url()` need no local work (their links just point at that URL); pages without one are rendered to a temp HTML mirror (same markdown→HTML pipeline as the answer), with their *own* cross-links resolved the same way — so a chain of links several hops deep out of the answer stays clickable, not just the first hop. Links to bundle paths the store doesn't have (not-yet-written concepts — allowed by the OKF convention) keep their original, already-inert bundle-relative href rather than being rewritten to a *new* guaranteed-dead one.

Rejected rendering the whole bundle only on the local backend and leaving xWiki links unrewritten (a per-backend `if`, not a `Store` method): would put backend branching back in `agent.py`, the exact fork ADR 0011 already ruled out for the file primitives. Rejected snapshotting xWiki pages to local HTML too (uniform look across backends): the snapshot goes stale the moment the live page changes, and duplicates rendering xWiki already does natively for free. Rejected rendering only the pages the answer directly links to (skip the whole-bundle walk): cheaper, but a link inside one of those pages to a second-hop page would still be dead — the whole point was that clicking through actually works.
